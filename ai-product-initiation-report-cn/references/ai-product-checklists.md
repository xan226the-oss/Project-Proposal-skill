# 人工智能产品立项检查清单

## 一、人工智能产品立项报告完整性检查清单

| 检查项 | 判断标准 |
|---|---|
| 项目概述 | 是否能在 1-2 分钟内说明项目价值、范围、资源、周期和结论 |
| 背景必要性 | 是否说明为什么现在必须做 |
| 目标用户 | 是否明确核心用户、次级用户、付费/管理用户 |
| 用户痛点 | 是否有场景、频率、严重程度和现有方案不足 |
| 人工智能价值 | 是否说明人工智能为什么比传统方案更适合 |
| 产品定位 | 是否一句话说明服务谁、解决什么问题、如何创造价值 |
| 最小可行产品范围 | 是否明确做什么、不做什么、验证什么 |
| 核心功能 | 是否每个功能都对应用户问题和验收标准 |
| 人工智能能力设计 | 是否说明输入、输出、模型/技术方案、评估指标和风险 |
| 数据方案 | 是否说明来源、权限、质量、更新、安全和闭环 |
| 技术方案 | 是否说明架构、系统流程、集成、部署和监控 |
| 成本预算 | 是否包含人力、模型、算力、数据、运维、推广、合规成本 |
| 商业价值 | 是否能用指标衡量增收、降本、提效、体验或竞争力 |
| 风险应对 | 是否包含需求、技术、数据、成本、产品、合规和商业风险 |
| 验收标准 | 是否包含产品、人工智能效果、业务和上线标准 |
| 立项结论 | 是否明确给出建议立项、最小可行产品验证、补充调研或暂不立项 |
| 信息标注 | 是否区分已知信息、假设、待补充、需验证 |
| 数据真实性 | 是否避免编造用户数、市场规模、预算、投资回报率、模型准确率 |

## 二、用户需求检查清单

- 是否明确目标用户角色。
- 是否说明用户在什么场景下遇到问题。
- 是否说明问题发生频率。
- 是否说明问题严重程度。
- 是否说明用户当前如何解决。
- 是否说明当前方案为什么不够好。
- 是否有用户访谈、业务反馈、数据指标或客户需求来源。
- 是否区分真实需求、管理诉求、技术尝试和主观判断。
- 是否说明用户是否愿意改变原流程。
- 是否说明用户是否有付费、使用或推广动机。

## 三、人工智能必要性检查清单

- 是否说明必须使用人工智能的原因。
- 是否说明传统规则、搜索、流程自动化或人工方案的不足。
- 是否明确人工智能介入点。
- 是否说明人工智能在理解、生成、检索、推荐、分析、自动化或多模态方面的价值。
- 是否说明人工智能输出如何被用户使用。
- 是否说明人工智能失败时的兜底流程。
- 是否说明人工智能能力边界。
- 是否避免只写“接入大模型”。

## 四、最小可行产品范围检查清单

- 是否明确第一阶段必须完成的核心场景。
- 是否明确暂时不做的能力。
- 是否明确后续迭代能力。
- 是否说明最小可行产品验证的核心假设。
- 是否避免功能范围过大。
- 是否能在既定资源和周期内交付。
- 是否有最小可用闭环。
- 是否有灰度用户或种子用户。
- 是否有上线后数据监控方案。

## 五、人工智能能力设计检查清单

- 是否列出人工智能能力类型，如问答、生成、抽取、总结、分类、推荐、检索增强生成、智能体、数据分析、多模态等。
- 是否说明每项能力的应用场景。
- 是否说明输入和输出。
- 是否说明模型方案：通用模型、垂直模型、自研模型、第三方接口或混合方案。
- 是否说明是否需要检索增强生成、向量数据库、知识库、微调、工具调用或智能体编排。
- 是否说明提示词与工作流策略。
- 是否说明上下文管理、多轮对话和用户反馈机制。
- 是否说明人工审核、人工确认或人工兜底。
- 是否说明幻觉、偏见、不可解释、误执行等风险。
- 是否说明离线评测和在线评测方法。

## 六、数据准备检查清单

- 是否明确数据来源。
- 是否明确数据类型。
- 是否明确数据用途。
- 是否评估数据质量。
- 是否明确数据权限。
- 是否说明数据更新频率。
- 是否说明数据清洗和标注需求。
- 是否说明隐私保护和安全策略。
- 是否说明数据是否可用于训练、微调或评测。
- 是否说明知识库维护机制。
- 是否说明数据闭环和用户反馈沉淀方式。

## 七、模型评估检查清单

- 是否定义准确率、召回率、幻觉率、任务成功率、响应时延、满意度等指标。
- 是否有离线测试集或评测样本。
- 是否有人工评审标准。
- 是否有线上反馈机制。
- 是否需要对照实验。
- 是否说明模型输出的可接受阈值。
- 是否说明错误案例分析机制。
- 是否说明模型升级和回归测试机制。
- 是否说明高风险输出的人工审核机制。

## 八、成本预算检查清单

- 是否包含研发人力成本。
- 是否包含模型调用成本。
- 是否包含算力或图形处理器成本。
- 是否包含数据采集、清洗和标注成本。
- 是否包含第三方服务成本。
- 是否包含测试评估成本。
- 是否包含运维、监控和日志成本。
- 是否包含市场推广或客户成功成本。
- 是否包含合规、安全和法务成本。
- 是否说明成本计算依据。
- 是否说明成本控制策略，如缓存、模型分层、小模型替代、提示词优化、批处理、限流和用户分级。

## 九、商业价值检查清单

- 是否说明用户价值。
- 是否说明业务价值。
- 是否区分增收、降本、提效、体验改善和竞争壁垒。
- 是否定义衡量指标。
- 是否说明商业模式，如订阅、按量计费、企业授权、私有化部署、接口收费或增值服务。
- 是否给出投资回报率公式或测算口径。
- 是否避免编造收入、成本和投资回报率。
- 是否说明价值验证周期。
- 是否说明未达预期时的调整策略。

## 十、风险合规检查清单

- 是否包含用户需求风险。
- 是否包含技术风险。
- 是否包含数据风险。
- 是否包含成本风险。
- 是否包含产品范围风险。
- 是否包含合规与安全风险。
- 是否包含商业风险。
- 是否说明风险等级、影响、概率、应对措施和责任方。
- 是否对隐私、版权、内容安全、数据跨境、高风险自动化决策进行提示。
- 是否对医疗、金融、教育、法律、招聘、风控、内容审核、未成年人等场景要求合规评估。
- 是否说明人工审核、审计记录和回滚机制。

## 十一、上会评审检查清单

- 是否有一页管理层摘要。
- 是否能清楚回答为什么现在做。
- 是否能说明投入多少资源。
- 是否能说明多久看到结果。
- 是否能说明成败如何衡量。
- 是否能说明最大风险和应对方式。
- 是否有明确立项建议。
- 是否列出需要评审会决策的问题。
- 是否列出前置条件。
- 是否有下一步计划。

## 十二、立项结论检查清单

| 结论 | 适用条件 |
|---|---|
| 建议立项 | 需求明确、价值清楚、技术路径可行、资源可获得、风险可控 |
| 建议先小范围验证最小可行产品后立项 | 方向有机会，但数据、模型效果、商业价值或用户需求仍需验证 |
| 建议补充调研后再决策 | 用户需求、市场空间、预算、技术方案或合规要求不清楚 |
| 暂不建议立项 | 价值不清、投入过高、风险不可控、缺乏真实用户需求或不具备实施条件 |

结论必须包含：

- 总体判断
- 主要理由
- 前置条件
- 下一步建议
